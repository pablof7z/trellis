The missing middle

Most reactive systems stop at value invalidation. A changed. B depends on A. Recompute B. Real application kernels often need more.

Value invalidation

A changed. B depends on A. Recompute B.

Resource reconciliation

A changed. B derived a new set. That set shrank. Old resources must close, new resources must open, stale output must clear, and the next visible state must be coherent.

Input->Derived Value->Collection Diff->Resource Plan->Materialized Output->Revision

Trellis turns dependency changes into inspectable resource plans, not hidden side effects.

A realistic example

Start with local-first sync, not Nostr. The reusable pattern is broader than any one protocol.

Local-first sync

active workspace -> visible projects -> sync windows -> replica demand -> issue board

When the user switches workspaces, Trellis can derive the new project set, close obsolete sync windows, open new ones, replay cached rows, clear stale board rows, and emit a new output revision.

Language servers

file changes -> module graph -> diagnostics -> semantic tokens -> editor output

When a file is deleted, stale diagnostics clear. When imports change, dependents invalidate. When a workspace closes, watchers and analysis tasks stop.

IoT dashboards

selected site -> visible devices -> topic subscriptions -> telemetry cards

When a filter shrinks, topic subscriptions shrink. Empty device sets subscribe to nothing, not accidentally broader demand.

Core model

Trellis gives names to the pieces most systems already hand-roll.

InputNode<T>

Canonical state changed by one owner.

DerivedNode<T>

Deterministic value derived from explicit dependencies.

CollectionNode<K, V>

A derived set or map with structural diffs.

ResourcePlan<C>

Commands produced from changes. The graph does not run arbitrary I/O.

Scope

A lifetime owner for resources, outputs, and dependencies.

MaterializedOutput<O>

A revisioned output surface with baseline, delta, clear, and rebaseline semantics.

Why plans, not effects?

Trellis does not encourage arbitrary effect closures that run because they happened to read a value. Propagation produces data: close this, open that, clear this output, emit this revision.

let demand = graph.resource_plan(
    "visible_device_topics",
    visible_devices.diff(),
    |diff| {
        Plan::new()
            .close_topics(diff.removed())
            .open_topics(diff.added())
            .clear_rows(diff.removed())
    },
);

The result is inspectable, testable, replayable, and easier to audit.

Built-in invariants

Source shrink withdraws resources

Removed collection members produce close commands.

Empty sources fail closed

Empty demand stays empty unless a fallback is explicit.

Scope close releases ownership

Closing a scope removes its resources and clears or rebaselines outputs.

Output is revisioned

Consumers receive coherent baseline, delta, clear, and rebaseline frames.

Incremental equals truth

Full recompute oracles can check incremental graph state.

Effects are planned first

The host actor applies commands after deterministic propagation.

Not another signal library

Use UI signals for UI state. Use Trellis when a derived value owns external resources.

Signals

Great for values and views. Not enough for resource lifecycle.

Rx

Great for stream composition. Not enough for scoped resource reconciliation.

Salsa

Great for pure incremental queries. Not enough for live resource plans and materialized output.

Trellis

Built for deterministic state-derived resource lifecycle.

0.1 status

Trellis 0.1 is intentionally narrow. The goal is not to solve all reactivity. The goal is to prove one reusable kernel.

Included

  • explicit nodes
  • collection diffs
  • resource plans
  • scopes
  • transactions
  • revisioned outputs
  • testing hooks

Not included

  • UI bindings
  • automatic dependency discovery
  • distributed execution
  • database persistence
  • domain-specific adapters
  • production stability guarantees

Trellis is for Rust developers who have hand-rolled callback glue to keep derived state, subscriptions, jobs, caches, and output surfaces in sync.
